QNX Vehicle Resource Manager

A full-stack embedded demo that connects a QNX Neutrino RTOS resource manager (the vehicle ECU side) to an Android Automotive OS VHAL bridge (the IVI head-unit side) over a TCP socket. It simulates CAN bus vehicle data — speed, gear, oil temperature, door lock — and exposes them both as UNIX-style device files on QNX and as live VHAL properties on Android.

What each layer does

CAN SimulatorGenerates realistic vehicle telemetry on a 100 ms timer. Replaces a real CAN transceiver + socketCAN read() on hardware.
Resource ManagerRegisters /dev/vehicle/* device nodes with the QNX Pathname Space. Responds to POSIX read() calls from any process using QNX IPC (MsgReceive).
Vehicle BridgeTCP client that serialises VehicleMessage structs and pushes them to the Android emulator. Handles reconnect automatically.
NDK Bridge (C++)TCP server inside Android. Deserialises structs and calls back into Java via JNI to keep Android state in sync.
VHAL Bridge ServiceAndroid Service that receives JNI callbacks and stores values in VhalStore. In a real AAOS build, calls CarPropertyManager instead.

Data flow — step by step

  1. main() calls can_sim_init() — sets initial vehicle state (speed=0, gear=PARK, oil=20°C, door=LOCKED).
  2. can_tick_thread calls can_sim_tick() every 100 ms and gets back a VehicleMessage struct plus the changed property ID.
  3. If the property changed, bridge_send() serialises the struct over TCP to 10.0.2.2:9090.
  4. On the Android side, bridge_thread (inside vhal_bridge.cpp) calls recv() for exactly 20 bytes, checks magic == 0xCAFEBABE, then dispatches to the correct JNI callback.
  5. The Java service writes to VhalStore. In a real AAOS build it calls CarPropertyManager.set*().
  6. Meanwhile, any process on QNX that opens /dev/vehicle/speed gets a formatted ASCII string from the resmgr io_read handler — no separate IPC needed.

Concepts & Patterns

This project is a compact demonstration of five distinct systems-programming disciplines stitched together. Here is a reference you can return to for any future project in this domain.

1. QNX Resource Manager Pattern

A resource manager is a user-space process that registers a pathname (e.g. /dev/vehicle/speed) with the QNX Pathname Space and then handles POSIX file calls (open, read, write, close) from other processes via QNX message passing.

It is the QNX equivalent of a Linux kernel character device driver — but it runs entirely in user space, is independently restartable, and cannot crash the kernel.

Key insight: On QNX every I/O call becomes an IPC message. The kernel routes read("/dev/vehicle/speed") to whichever process registered that path. MsgReceive ↔ MsgReply is the fundamental primitive.

The dispatch loop

dispatch_t *dpp = dispatch_create();      // create message channel
resmgr_attach(dpp, &rattr, path, ...);   // publish path to namespace
dispatch_context_t *ctp = dispatch_context_alloc(dpp);

while (1) {
    ctp = dispatch_block(ctp);            // blocks on MsgReceive
    dispatch_handler(ctp);                // calls the registered io_funcs
}

io_read handler contract

static int io_read(resmgr_context_t *ctp, io_read_t *msg, RESMGR_OCB_T *ocb) {
    if (ocb->offset > 0) return _RESMGR_NPARTS(0);  // EOF on 2nd read
    SETIOV(ctp->iov, buf, len);                       // set reply buffer
    ocb->offset += len;
    return _RESMGR_NPARTS(1);                         // 1 iov part to send
}

RESMGR_OCB_T = Open Control Block — one per open() call. ocb->offset tracks how many bytes have been sent so sequential reads work correctly.

VehicleAttr — extending iofunc_attr_t

typedef struct {
    iofunc_attr_t attr;     // MUST be first member — QNX casts VehicleAttr* to iofunc_attr_t*
    uint32_t      prop_id;  // which vehicle property this node serves
    char          path[64];
} VehicleAttr;

This is the struct inheritance idiom in C: by placing the base struct first, a pointer to VehicleAttr is safely cast to iofunc_attr_t * by the QNX resmgr library.

2. CAN Bus Simulation Pattern

CAN (Controller Area Network) is the dominant in-vehicle serial bus. ECUs broadcast fixed-size frames (8 bytes data + 11-bit or 29-bit message ID) at rates from 10 Hz to 1 kHz. The QNX kernel exposes a physical CAN controller as /dev/can0; you read frames with read(fd, &frame, sizeof(frame)).

This project replaces the hardware with a state-machine simulator driven by a monotonic tick counter. The pattern is: 

// Real hardware:
fd = open("/dev/can0", O_RDWR);
read(fd, &frame, sizeof(struct can_frame));  // blocks until frame arrives

// Simulator:
uint32_t can_sim_tick(VehicleMessage *out) {
    static int tick = 0; tick++;
    switch ((tick / 10) % 4) {  // rotate through 4 properties
    case 0: /* update speed  */ return PROP_VEHICLE_SPEED;
    case 1: /* update gear   */ return PROP_GEAR_SELECTION;
    ...
    }
}
Pattern — Software CAN simulation: Divide tick counter by the number of ticks per update period, then modulo by the number of properties to get a round-robin property update loop that is deterministic and testable.

Monotonic timestamp

static int64_t monotonic_ms(void) {
    struct timespec ts;
    clock_gettime(CLOCK_MONOTONIC, &ts);
    return (int64_t)ts.tv_sec * 1000 + ts.tv_nsec / 1000000;
}

CLOCK_MONOTONIC never jumps backwards (unlike wall clock). Always use it for delta timing in embedded code.

3. Binary TCP Protocol Pattern

Instead of JSON or text, the bridge sends a packed C struct directly over a TCP stream. This is common in embedded/automotive middleware because it is zero-copy and has predictable latency. 

typedef struct __attribute__((packed)) {
    uint32_t   magic;        // 0xCAFEBABE — sanity / frame guard
    uint32_t   prop_id;      // which property
    PropValue  value;        // union: float / int32 / uint8
    int64_t    timestamp_ms; // QNX monotonic ms
} VehicleMessage;            // total: 4+4+4+8 = 20 bytes (no padding)
Endianness note: Both sides must be the same endianness for this to work correctly without byte-swapping. In production, use htonl/ntohl or define an explicit wire format.

Sender (QNX)

send(s_sockfd, msg, sizeof(*msg), MSG_NOSIGNAL);
// MSG_NOSIGNAL: don't raise SIGPIPE if the receiver closed — return EPIPE instead

Receiver (Android NDK)

recv(client, &msg, sizeof(msg), MSG_WAITALL);
// MSG_WAITALL: block until ALL sizeof(msg) bytes arrive (not a short read)

Reconnect pattern

int bridge_send(const VehicleMessage *msg) {
    if (s_sockfd < 0) {
        if (bridge_connect() < 0) return -1;  // best-effort
    }
    ssize_t n = send(s_sockfd, msg, sizeof(*msg), MSG_NOSIGNAL);
    if (n != sizeof(*msg)) {
        close(s_sockfd); s_sockfd = -1;   // mark disconnected
        return -1;
    }
    return 0;
}

The function keeps a file-descriptor as module-level state (static int s_sockfd = -1). -1 is the sentinel for "not connected." Each send attempt auto-reconnects.

4. JNI Bidirectional Bridge Pattern

JNI (Java Native Interface) is the Android mechanism that lets C/C++ code call Java methods and vice versa. The pattern used here is: Java calls a native method to start a C thread, and that C thread later calls Java methods (callbacks) via stored method IDs.

Step 1 — Java declares the native method

private native void startBridge();  // implemented in vhal_bridge.cpp

Step 2 — JNI_OnLoad captures the JavaVM

JNIEXPORT jint JNI_OnLoad(JavaVM *vm, void *) {
    s_jvm = vm;          // save JavaVM* for later use on background threads
    return JNI_VERSION_1_6;
}

Step 3 — startBridge resolves method IDs and spawns thread

s_callback = env->NewGlobalRef(thiz);              // keep Java object alive
jclass cls = env->GetObjectClass(thiz);
s_onPropF  = env->GetMethodID(cls, "onFloatProperty", "(IF)V");  // JNI signature
s_onPropI  = env->GetMethodID(cls, "onIntProperty",   "(II)V");
s_onPropB  = env->GetMethodID(cls, "onBoolProperty",  "(IZ)V");
pthread_create(&tid, nullptr, bridge_thread, nullptr);

Step 4 — Background thread attaches to JVM and calls back

JNIEnv *env = nullptr;
s_jvm->AttachCurrentThread(&env, nullptr);  // required for non-Java threads
// ... receive data ...
env->CallVoidMethod(s_callback, s_onPropF, (jint)propId, (jfloat)value);
s_jvm->DetachCurrentThread();               // clean up on exit
JNI type signatures: "(IF)V" = takes int, float, returns void. I=int, F=float, Z=boolean, V=void.

5. Dual-Target Build Pattern (Makefile)

A single codebase must build for two targets: the actual QNX cross-compiler (qcc) and a Linux host GCC build for CI. The pattern uses a HOST variable: 

ifndef HOST
  CC     := qcc -Vgcc_ntox86_64   # QNX cross-compiler
  CFLAGS := -D_QNX_SOURCE ...
else
  CC     := gcc
  CFLAGS := -DHOST_BUILD -I./qnx_stubs ...  # mock QNX headers
endif

qnx_stubs/ contains header shims (sys/iofunc.h, sys/dispatch.h) that define the QNX types as no-ops so the code compiles on Linux for unit-testing the pure logic (CAN simulator, bridge socket) without a QNX license.

6. Singleton + ConcurrentHashMap Pattern (VhalStore)

private static final VhalStore INSTANCE = new VhalStore();
public static VhalStore getInstance() { return INSTANCE; }

private final ConcurrentHashMap<Integer, Object> mProps = new ConcurrentHashMap<>();

ConcurrentHashMap provides lock-free reads and fine-grained locking on writes — safe for concurrent access from the native bridge thread (writing) and any Android UI thread (reading). The key is the int property ID; the value is boxed as Object to hold any of Float, Integer, or Boolean. 

// Read with type check and default:
public float getFloat(int propId, float def) {
    Object v = mProps.get(propId);
    return v instanceof Float ? (float)v : def;
}

7. Minimal Unit Test Framework (C, no external lib)

Rather than linking gtest or Unity, the test file defines its own three-macro framework:

#define TEST(name)   printf("  %-45s", name)
#define PASS()       do { printf("PASS\n"); s_pass++; } while(0)
#define FAIL(msg)    do { printf("FAIL — %s\n", msg); s_fail++; } while(0)
#define ASSERT_EQ(a,b,msg) do { if((a)==(b)) PASS(); else FAIL(msg); } while(0)
#define ASSERT_GT(a,b,msg) do { if((a)> (b)) PASS(); else FAIL(msg); } while(0)

The do { … } while(0) idiom makes macros safe inside if statements without braces. The test runner exits with EXIT_FAILURE if any test failed, enabling CI to catch regressions.

function can_sim_tick()

Contains the main simulation loop / switch:

uint32_t can_sim_tick(VehicleMessage *out) {
    static int tick = 0;   // persists across calls (function-local static)
    tick++;

    out->magic        = BRIDGE_MAGIC;
    out->timestamp_ms = monotonic_ms();

    switch ((tick / 10) % 4) {   // rotate every 10 ticks, 4 phases

    case 0:  // SPEED — ramps 0→120, wraps back to 0
        s_speed_kmh += 2.0f;
        if (s_speed_kmh > 120.0f) s_speed_kmh = 0.0f;
        s_gear = (s_speed_kmh > 5.0f) ? GEAR_DRIVE : GEAR_PARK;
        out->prop_id = PROP_VEHICLE_SPEED;
        out->value.f = s_speed_kmh;
        return PROP_VEHICLE_SPEED;

    case 1:  // GEAR — derived from speed
        out->prop_id = PROP_GEAR_SELECTION;
        out->value.i = s_gear;
        return PROP_GEAR_SELECTION;

    case 2:  // OIL TEMP — warms up to 90 °C, clamps
        s_oil_temp_c += 0.5f;
        if (s_oil_temp_c > 90.0f) s_oil_temp_c = 90.0f;
        out->prop_id = PROP_ENGINE_OIL_TEMP;
        out->value.f = s_oil_temp_c;
        return PROP_ENGINE_OIL_TEMP;

    case 3:  // DOOR LOCK — toggles every 40 ticks
        if (tick % 40 == 0) s_door_locked ^= 1;   // XOR toggle
        out->prop_id = PROP_DOOR_LOCK;
        out->value.b = s_door_locked;
        return PROP_DOOR_LOCK;

    default: return 0;
    }
}

Note the static int tick = 0 inside the function — this is a function-local static, initialised once and persistent across calls, without polluting file scope.

notable MSG_NOSIGNAL flag

send(s_sockfd, msg, sizeof(*msg), MSG_NOSIGNAL);

Without this flag, writing to a broken TCP connection raises SIGPIPE, which by default kills the process. MSG_NOSIGNAL converts that to a return of -1 with errno == EPIPE so the caller can handle it gracefully.

function format_prop()

static int format_prop(uint32_t prop_id, char *buf, size_t len) {
    switch (prop_id) {
    case PROP_VEHICLE_SPEED:   return snprintf(buf, len, "%.2f km/h\n", can_sim_get_speed());
    case PROP_GEAR_SELECTION:  /* nested switch on gear value */ ...
    case PROP_ENGINE_OIL_TEMP: return snprintf(buf, len, "%.1f C\n", can_sim_get_oil_temp());
    case PROP_DOOR_LOCK:       return snprintf(buf, len, "%s\n", ...);
    }
}

Uses snprintf (not sprintf) for buffer safety. Returns the number of bytes written (excluding null terminator), matching the POSIX read() convention.

function io_read() — the core handler

static int io_read(resmgr_context_t *ctp, io_read_t *msg, RESMGR_OCB_T *ocb) {
    VehicleAttr *attr = (VehicleAttr *)ocb->attr;  // cast base to derived
    char buf[64];

    if (ocb->offset > 0) return _RESMGR_NPARTS(0); // EOF on re-read

    int len = format_prop(attr->prop_id, buf, sizeof(buf));
    SETIOV(ctp->iov, buf, len);  // set reply iov (scatter-gather)
    ocb->offset += len;
    return _RESMGR_NPARTS(1);    // 1 iov segment in reply
}

function register_device()

static void register_device(dispatch_t *dpp, resmgr_attr_t *rattr,
                             iofunc_funcs_t *funcs, resmgr_io_funcs_t *io_funcs,
                             VehicleAttr *dev, const char *path, uint32_t prop_id) {
    iofunc_attr_init(&dev->attr, S_IFCHR | 0444, NULL, NULL); // char device, read-only
    dev->prop_id = prop_id;
    strncpy(dev->path, path, sizeof(dev->path) - 1);

    resmgr_attach(dpp, rattr, path, _FTYPE_ANY, 0, io_funcs, funcs, &dev->attr);
}

S_IFCHR | 0444 = character device, world-readable. _FTYPE_ANY = accept any open type (regular, block, etc.).

thread can_tick_thread()

static void *can_tick_thread(void *arg) {
    VehicleMessage msg;
    bridge_connect();   // best-effort; bridge_send retries on failure

    while (1) {                              // infinite producer loop
        uint32_t changed = can_sim_tick(&msg);
        if (changed) bridge_send(&msg);
        usleep(100 * 1000);                  // 100 ms = 10 Hz
    }
    return NULL;
}

Pattern: producer thread with sleep-based rate limiting. usleep(100000) keeps the loop at exactly 10 Hz — matching a real CAN bus update rate for these properties.

function main()

Sequence of setup calls:

  1. can_sim_init() — reset simulator state
  2. dispatch_create() — create QNX message channel
  3. iofunc_func_init() — fill io_funcs with QNX defaults
  4. Override io_funcs.read and io_funcs.write with custom handlers
  5. mkdir("/dev/vehicle", 0755) — ensure directory exists
  6. register_device() ×4 — publish each /dev/vehicle/* node
  7. pthread_create(&tick_tid, …, can_tick_thread, …) — start CAN loop
  8. dispatch_context_alloc() + dispatch_block() loop — enter message receive loop

thread bridge_thread() — two nested loops

while (true) {                              // outer: accept loop
    int client = accept(server_fd, ...);
    LOGI("QNX connected");

    VehicleMessage msg;
    while (true) {                          // inner: receive loop
        ssize_t n = recv(client, &msg, sizeof(msg), MSG_WAITALL);
        if (n != sizeof(msg)) break;        // disconnect → break to outer
        if (msg.magic != BRIDGE_MAGIC) continue;  // bad frame → skip

        switch (msg.prop_id) {              // dispatch by type
        case PROP_VEHICLE_SPEED:
        case PROP_ENGINE_OIL_TEMP:
            env->CallVoidMethod(s_callback, s_onPropF, ...); break;
        case PROP_GEAR_SELECTION:
            env->CallVoidMethod(s_callback, s_onPropI, ...); break;
        case PROP_DOOR_LOCK:
            env->CallVoidMethod(s_callback, s_onPropB, ...); break;
        }
    }
    close(client);
}

The outer loop allows reconnection from QNX — if QNX restarts, the Android side automatically accepts the new connection without restarting.

}

Automotive Stack Explained

This project touches four layers of the automotive software stack. Here is a reference for each layer from hardware up to the application.

Layer 0 — CAN Bus (physical / data link)

Controller Area Network is a two-wire differential serial bus invented by Bosch in 1983, now mandatory in every road vehicle (ISO 11898). Every node (ECU) broadcasts messages; every node receives every message. There is no master/slave — the bus is multi-master with priority arbitration.

PropertyClassic CAN (2.0)CAN FD
Bit rateup to 1 Mbit/sup to 8 Mbit/s (data phase)
Data bytes0–8 bytes0–64 bytes
Message ID11-bit (2.0A) or 29-bit (2.0B)same, extended
Error detectionCRC + ACK + bit stuffingsame + SECDED in FD

Typical CAN frame

struct can_frame {
    canid_t  can_id;    // 11 or 29-bit message ID
    uint8_t  can_dlc;  // data length code (0–8)
    uint8_t  data[8];  // payload
};

Common CAN message IDs in a vehicle

ID (hex)SignalRate
0x0C0Engine speed (RPM)10 ms
0x0D0Vehicle speed10 ms
0x100Gear position100 ms
0x420Door statuson-change
0x640Oil temperature1000 ms

socketCAN — Linux / QNX kernel interface

// Real hardware — read a raw CAN frame:
int fd = open("/dev/can0", O_RDWR);    // QNX
// or: socket(PF_CAN, SOCK_RAW, CAN_RAW);  // Linux socketCAN

struct can_frame frame;
read(fd, &frame, sizeof(frame));
uint16_t speed_raw = (frame.data[0] << 8) | frame.data[1];
float speed_kmh = speed_raw * 0.01f;   // decode per DBC file

A DBC file (Database CAN) is a text file that maps CAN IDs + bit positions to engineering-unit signal names. Tools like Vector CANdb++, cantools (Python), and kvaser.com parse DBC files.

Layer 1 — QNX Neutrino RTOS

QNX Neutrino is a microkernel RTOS. The kernel only does scheduling, IPC, and interrupt dispatch. All drivers, filesystems, and network stacks run as user-space processes communicating via message passing. A crashed driver is simply restarted — the kernel does not crash.

Key QNX concepts

ConceptMeaning
Pathname SpaceGlobal namespace where both file paths and service endpoints live. resmgr_attach() inserts a path into it.
MsgSend / MsgReceive / MsgReplyThe kernel IPC primitives. Every POSIX call to a device becomes a MsgSend to the owning process.
PulseSmall fixed-size IPC message for lightweight signals (e.g. timer expiry).
dispatch_tMultiplexes a single MsgReceive channel across multiple resource manager paths.
OCB (Open Control Block)Created per open() call, destroyed on close(). Carries file position, flags, per-client state.
iofunc_attr_tPer-device attributes: mode, owner, size, timestamps. The QNX equivalent of Linux struct inode.

QNX in the automotive stack

QNX powers the instrument cluster, ADAS domain controller, and digital cockpit in vehicles from BMW, Ford, GM, Volkswagen, and others. It provides hard real-time guarantees (microsecond-precision scheduling) and functional safety certification up to ISO 26262 ASIL D.

QNX Hypervisor (QNX SDP 8 / Qualcomm SA8155/SA8295)

On multi-core SoCs like the Qualcomm SA8155P (Snapdragon Automotive), QNX runs as a Type-1 hypervisor partitioning CPUs between:

  • QNX guest VM — real-time cluster / ADAS / safety
  • Android AAOS guest VM — IVI head unit, maps, media

The two VMs communicate via virtio channels or shared memory (qvm_vdev_shmem). This project uses a TCP socket as the equivalent for emulator development.

Layer 2 — Android Automotive OS (AAOS) & VHAL

Android Automotive OS is a full Android build customised for in-vehicle infotainment. It adds the Vehicle Hardware Abstraction Layer (VHAL) — the interface between the Android car framework and the actual vehicle hardware (ECUs, gateways).

Layer 3 — Android NDK & JNI

The Android NDK lets you write C/C++ code that is compiled to a native shared library (.so) and loaded into an Android process via System.loadLibrary(). JNI is the ABI that lets C code call Java and Java call C.

Why NDK for this bridge?

  • Java can't do recv(MSG_WAITALL) on a raw socket with zero-copy semantics efficiently.
  • The binary protocol struct must be deserialised exactly — C/C++ handles packed struct layout deterministically.
  • NDK threads can attach to the JVM and call back into Java, enabling a clean producer-consumer pattern.

/* Replaces vehicle_bridge.c on i.MX 8QM */
#include <rpmsg_lite.h>

static struct rpmsg_lite_instance *s_rpmsg = NULL;
static struct rpmsg_lite_endpoint *s_ept   = NULL;

int bridge_connect(void) {
    s_rpmsg = rpmsg_lite_master_init(RPMSG_SHMEM_BASE, RPMSG_SHMEM_SIZE,
                                      RPMSG_LITE_LINK_ID, RL_NO_FLAGS);
    s_ept   = rpmsg_lite_create_ept(s_rpmsg, LOCAL_EPT_ADDR,
                                    rpmsg_queue_rx_cb, NULL);
    return s_rpmsg ? 0 : -1;
}

int bridge_send(const VehicleMessage *msg) {
    return rpmsg_lite_send(s_rpmsg, s_ept, REMOTE_EPT_ADDR,
                           (char *)msg, sizeof(*msg), RL_BLOCK);
}
